# Hybrid Contract Search Rollout Plan

This document breaks the architecture work into sequential phases so implementation can proceed safely and be validated step by step.

Rule of execution:
- Do not start the next phase until the current phase works and has at least one basic verification check.
- Do not hand the whole architecture brief to an AI agent at once.
- Feed the agent only the section relevant to the current phase.

## Working Rules

### 1. Scope control
- Give the agent only the current phase requirements, not the full architecture document.
- Keep each implementation step narrow enough that it can be reviewed manually.
- Prefer small, reversible changes over large cross-cutting rewrites.

### 2. Logging
- Require structured logging in worker and ingestion code.
- Log document status transitions explicitly.
- Log Celery task start, retry, failure, and completion events.
- Log external integration failures separately for database, PDF parsing, LLM, and Qdrant.

### 3. Migration safety
- Review Alembic migrations manually before applying them.
- Pay special attention to destructive operations such as `drop table`, `drop column`, or type changes.
- Do not trust autogenerated migration SQL without review.

## Phase 1: Foundation

Goal:
- Create the PostgreSQL schema and the basic upload/status API without vectors or LLM retrieval.

Deliverables:
- SQLAlchemy models
- Alembic migrations
- upload endpoint
- status endpoint

### Phase 1.1: Database schema
Relevant architecture slice:
- Storage model, PostgreSQL only

Agent prompt:
> Write SQLAlchemy models and Alembic migrations for `documents`, `contract_facts` using `JSONB`, and `extraction_runs`. Do not implement anything about vectors or retrieval.

Expected outcome:
- `documents` stores file metadata, lifecycle status, processing status, active extraction version, and last error
- `contract_facts` stores extracted JSON facts per document
- `extraction_runs` stores processing attempts and version metadata

Basic checks:
- migration upgrades successfully
- tables exist with expected columns
- rollback path is valid

### Phase 1.2: Upload API
Relevant architecture slice:
- Asynchronous ingestion pipeline, upload behavior only

Agent prompt:
> Implement `POST /documents/upload` in FastAPI. It must only save the file, create a database record with `PENDING_REVIEW` and `QUEUED`, and return `200 OK`. Mock asynchronous work for now with a placeholder function.

Expected outcome:
- request returns quickly
- no parsing or LLM work happens inline
- document row is created with the correct initial statuses

Basic checks:
- upload returns `200 OK`
- file is persisted
- database row is created with expected status values

### Phase 1.3: Status API
Agent prompt:
> Implement `GET /documents/{id}` that returns the current document statuses and extracted facts from the database.

Expected outcome:
- frontend can poll document status
- API returns facts when available and error state when processing failed

Basic checks:
- endpoint returns document metadata
- endpoint includes facts if they exist
- endpoint returns a clear error for missing document ids

Stop condition for Phase 1:
- Upload works end to end without timeout
- Status polling works
- No vector or LLM functionality is required yet

## Phase 2: Background Processing

Goal:
- Move heavy PDF processing into a durable worker and produce reviewed facts without indexing.

Deliverables:
- Celery + Redis wiring
- layout analysis service
- fact extraction service
- status transitions through background jobs

### Phase 2.1: Celery infrastructure
Agent prompt:
> Configure Celery + Redis for FastAPI. Add a worker entrypoint. Replace the placeholder in `POST /documents/upload` with `process_document.delay(document_id)`.

Expected outcome:
- API enqueues a durable task
- worker can run independently
- failures and retries are visible in logs

Basic checks:
- upload enqueues a Celery task
- worker receives the task
- status moves from `QUEUED` to `PARSING`

### Phase 2.2: Layout analysis
Agent prompt:
> Write a service function that reads a PDF from disk, runs it through DocLing or an equivalent library, and returns Markdown that preserves tables. Integrate this into the Celery task and set the status to `PARSING`.

Expected outcome:
- long PDFs are processed in the worker, not in the API
- Markdown preserves headings, tables, and page boundaries as well as possible

Basic checks:
- worker loads a sample PDF
- output Markdown is persisted or traceable
- parsing failure updates status and logs an actionable error

### Phase 2.3: Fact extraction
Agent prompt:
> Write a fact extraction service. Use a Pydantic schema to validate the LLM response. Input is Markdown from DocLing. Output is JSON. Save the JSON into `contract_facts` in Postgres and move the status to `FACTS_READY`.

Expected outcome:
- extracted facts are validated before storage
- facts are saved without any vector indexing
- the document becomes ready for manual review

Basic checks:
- valid LLM output is stored in `contract_facts`
- invalid LLM output fails clearly and logs the validation problem
- status becomes `FACTS_READY` on success

Stop condition for Phase 2:
- Upload triggers background processing
- worker extracts facts and persists them
- document reaches `FACTS_READY` without any vector operations

## Phase 3: Approval and Vector Indexing

Goal:
- Add manual confirmation and index approved data into Qdrant.

Deliverables:
- confirmation endpoint
- Qdrant client
- summary and chunk collections
- idempotent indexing task

### Phase 3.1: Confirmation API
Agent prompt:
> Implement `POST /documents/{id}/confirm`. It must update JSON facts in the database if the manager edited them, set the document status to `APPROVED`, and trigger `index_document.delay(document_id)`.

Expected outcome:
- only approved facts can proceed to indexing
- document review is a real gating step

Basic checks:
- approved facts overwrite pending facts
- document status becomes `APPROVED`
- indexing task is enqueued

### Phase 3.2: Qdrant setup
Agent prompt:
> Write a Qdrant client class. Initialize two collections: `contract_summaries` and `contract_chunks` with appropriate vector settings.

Expected outcome:
- summaries and chunks are isolated in separate collections
- the client can create or verify collections safely

Basic checks:
- startup or indexing path ensures collections exist
- collection configuration is deterministic

### Phase 3.3: Idempotent indexing
Agent prompt:
> Implement `index_document`. Step 1: generate a summary through the LLM. Step 2: split the Markdown into chunks. Step 3: delete old vectors by `document_id`. Step 4: insert new vectors into both collections. Set the status to `INDEXED`.

Expected outcome:
- indexing uses approved facts only
- reprocessing does not create duplicate vectors
- summary and chunk collections are both refreshed

Basic checks:
- indexing writes into both collections
- rerunning indexing for the same document does not create duplicates
- failed indexing moves the document to a failure state with logs

Stop condition for Phase 3:
- approval is required before indexing
- both collections are populated correctly
- reindexing is safe

## Phase 4: Search Orchestration

Goal:
- Connect SQL and vector search through explicit query routing.

Deliverables:
- query router
- SQL search service
- hybrid vector search service
- chat orchestration that uses the selected route

### Phase 4.1: Query router
Relevant architecture slice:
- Query routing and retrieval

Agent prompt:
> Write a service that classifies a user query into one of five route types: `SQL_ONLY`, `SUMMARY_ONLY`, `SUMMARY_THEN_CHUNKS`, `CHUNKS_ONLY`, or `HYBRID_SQL_AND_CHUNKS`. Use either a lightweight LLM or deterministic rules. Return both the route type and extracted filters such as `year=2023`.

Expected outcome:
- route selection is explicit and testable
- the router can detect structured filters for SQL queries

Basic checks:
- list/report queries map to `SQL_ONLY`
- topic discovery queries map to `SUMMARY_ONLY`
- explanation queries map to `SUMMARY_THEN_CHUNKS`

### Phase 4.2: SQL search
Agent prompt:
> Write a `SQL_ONLY` search service that generates a SQLAlchemy query against `contract_facts` based on extracted filters.

Expected outcome:
- structured queries can be answered without vector search
- exact reporting avoids noisy chunk retrieval

Basic checks:
- queries filtered by year, date, or supplier return the expected rows
- empty result sets are handled cleanly

### Phase 4.3: Hybrid vector search
Agent prompt:
> Write a vector search service for `SUMMARY_THEN_CHUNKS`. First search `contract_summaries`. Take the top 3 `document_id` values. Then search `contract_chunks` filtered strictly to those 3 documents.

Expected outcome:
- global discovery is driven by summaries first
- chunk search is narrowed to relevant documents

Basic checks:
- summary search returns ranked documents
- chunk search is limited to summary-selected ids
- explanation-style answers use the routed evidence

Stop condition for Phase 4:
- chat path routes correctly
- SQL and vector paths both work
- hybrid search uses summary-first narrowing before chunk retrieval

## Definition of Done
- upload returns immediately
- heavy document processing happens only in the worker
- approved facts gate all indexing
- summaries and chunks are stored in separate Qdrant collections
- reprocessing deletes old vectors before inserting new ones
- retrieval is routed instead of always running every path
