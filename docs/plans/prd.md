## **1. Operational Context & Strategic Fit**

**Проблема:** Существующий процесс контроля за возвратом закрывающих документов (актов) опирается на человеческую память и ручной ввод. Это приводит к потере документов, просрочкам, юридическим рискам и неэффективной работе бухгалтерии.

**Решение:** Внедрение внутреннего продукта, где основным интерфейсом взаимодействия выступает не классическая таблица, а **AI-ассистент (RAG)**. Система берет на себя рутину по парсингу условий, контролю сроков и поиску информации, оставляя за менеджером только функцию валидации.

**ROI (Окупаемость):**

- **Снижение издержек:** Превентивное исключение просрочек по документам, защита от работы с недобросовестными контрагентами.
- **Экономия времени (Time-to-Action):** Сокращение времени на поиск юр. деталей и подбор подрядчиков за счет семантического поиска.

## **2. User Personas & JTBD**

- **Менеджер (Инициатор):** *«Я хочу загрузить договор, быстро подтвердить распознанный дедлайн и забыть о нем, зная, что система сама мягко напомнит мне об акте, а AI-чат поможет мгновенно найти детали прошлых сделок».*
- **Директор / Админ:** *«Я хочу видеть прозрачный дашборд с показателями дисциплины (кто из менеджеров копит просрочки), чтобы принимать управленческие решения на основе данных, а не эмоций».*

## **3. Core Product Mechanics: Интеллектуальный процесс**

Вместо классической CRM система строится вокруг четырех основ:

1. **AI-Chat UI как ядро:** Единое окно работы — интерфейс чата с историей диалогов.
    - *Глобальный контекст:* Поиск по всей базе через заранее подготовленные саммари документов (например, «найди проблемных поставщиков яблок»).
    - *Локальный контекст:* Работа с конкретной карточкой контрагента/договора, где поиск идет по текстовым чанкам выбранного документа, а не по извлеченному JSON (например, «какие условия расторжения в одностороннем порядке описаны в этом документе?»).
2. **Human-in-the-Loop (HitL) при загрузке:** Менеджер загружает PDF. AI-пайплайн автоматически извлекает ключевые метаданные и формирует черновые саммари, после чего менеджер визуально проверяет извлеченные JSON-факты и нажимает кнопку подтверждения. Только после ручного подтверждения данные считаются достоверными и могут быть записаны в поисковые индексы.
3. **Cron-Оповещения (Система светофора):** По загруженным договорам без актов (связь 1:1) работает фоновый планировщик, отправляющий email-напоминания:
    - 🟢 **Зеленая зона:** 15 дней до дедлайна.
    - 🟡 **Желтая зона:** 10 дней до дедлайна.
    - 🔴 **Красная зона:** 5 дней до дедлайна (напоминание только менеджеру).
    - ⚫ **Просрок:** Дедлайн истек. Выставляется системный флаг нарушения на основе SQL-логики (без участия LLM, без авто-эскалации, данные уходят на дашборд Директора).
4. **Executive Dashboard (Контур подотчетности):** Механизм гарантированного внедрения (Adoption) инструмента в бизнес-процесс. Дашборд руководителя замыкает петлю контроля, агрегируя данные о работе системы:
    - *Дисциплинарные метрики:* Визуализация доли договоров с флагом «Просрочено» в разрезе каждого менеджера для выявления узких мест.
    - *Контроль использования:* Отслеживание объема загружаемых контрактов, что исключает возможность саботажа нового процесса и гарантирует, что вся работа с подрядчиками ведется исключительно через RAG-систему.

## **4. Technical Architecture Constraints (Фундамент)**

- **Хранилище данных: трехуровневая модель**
    - **Слой A (SQL / PostgreSQL):** Хранение жестких фактов и бизнес-состояния. Сюда входят JSONB-поля с извлеченными атрибутами (ИНН, даты, суммы, статусы актов), связи между сущностями, а также ролевая модель доступа (RBAC) и ownership документов.
    - **Слой B (Qdrant / `contract_summaries`):** Хранение эмбеддингов кратких саммари по каждому документу. Слой используется для быстрого глобального семантического поиска по всей базе договоров и контрагентов.
    - **Слой C (Qdrant / `contract_chunks`):** Хранение детализированных чанков текста с привязкой к структуре документа (заголовки, разделы, абзацы, page anchors). Слой используется для глубокого поиска внутри конкретного договора.
- **Backend & Async Pipeline:** Основной backend реализуется на FastAPI. Тяжелые операции ingestion-пайплайна (DocLING Layout Analysis, OCR при необходимости, LLM Extraction, chunking, summary generation) выносятся в асинхронные воркеры на Celery + Redis, чтобы не блокировать интерфейс и упростить масштабирование.
- **Human-in-the-Loop как граница доверия:** Извлеченные факты сначала попадают в черновик проверки. Векторизация и запись в Qdrant допускаются только после ручного подтверждения менеджером в UI; это снижает риск галлюцинаций и не позволяет неподтвержденным данным влиять на поиск.
- **Query Router:** Интеллектуальный роутер на уровне backend определяет, какой слой нужен для ответа на запрос:
    - SQL-only для строгих фактов, дедлайнов, статусов и фильтров.
    - `contract_summaries` для глобального поиска по смыслу между документами.
    - `contract_chunks` для глубокого поиска внутри выбранного документа с жесткой фильтрацией по `document_id`.
- **Frontend UX:** Клиентский интерфейс на SvelteKit показывает результаты глобального поиска как список компаний/договоров с предсчитанными саммари, а локальный чат в карточке документа работает по найденным релевантным чанкам и возвращает ответ с цитатой.
- **Логгирование:** MVP реализуется через простой лог статусов документов. Архитектура таблиц проектируется с учетом будущего легкого перехода на событийную модель (Event Audit).

## **5. Validation of Key Use Cases**

### **5.1. Поиск поставщиков по промпту (Глобальный поиск)**

**Статус:** Реализуемо.

**Как это работает:** Менеджер вводит свободный запрос (например, «проблемные поставщики яблок»). Query Router классифицирует его как глобальный семантический поиск и направляет запрос в коллекцию `contract_summaries` в Qdrant.

**Результат:** Система находит наиболее релевантные саммари документов. По найденным `document_id` backend подтягивает из PostgreSQL метаданные поставщиков, юридические атрибуты и список всех связанных договоров.

**UX:** На стороне SvelteKit пользователь видит список компаний. При открытии карточки показываются заранее сохраненное саммари и перечень связанных контрактов, а не сгенерированный "на лету" пересказ.

### **5.2. Поиск внутри одного договора (Локальный поиск)**

**Статус:** Реализуемо.

**Ограничение:** JSON-факты из PostgreSQL подходят только для сухих бизнес-атрибутов (даты, суммы, стороны, статусы), но не позволяют отвечать на вопросы по сложным правовым формулировкам.

**Как это работает:** Когда пользователь открывает договор и задает вопрос в локальном чате, Query Router использует слой `contract_chunks`. В запрос добавляется жесткий фильтр `document_id = X`, после чего поиск выполняется только по чанкам выбранного договора.

**Результат:** Система находит релевантные абзацы и передает их в LLM как контекст для ответа. Пользователь получает краткий ответ с опорой на конкретную цитату или выдержку из документа.

### **5.3. Уведомления по актам (Cron)**

**Статус:** Реализуемо и является наиболее надежным сценарием MVP.

**Как это работает:** Для уведомлений не используются ни Qdrant, ни LLM. PostgreSQL хранит поля `act_deadline` и `act_received_status`, а планировщик Celery Beat раз в сутки выполняет SQL-запрос по документам, где акт не получен и до дедлайна остается 15, 10 или 5 дней.

**Результат:** По найденным записям формируется очередь на отправку email-уведомлений. Переход в статус "Просрочено" и отображение на дашборде директора строятся на той же SQL-логике, что делает сценарий предсказуемым и проверяемым.

## **6. Roadmap: Уточненные фазы реализации**

Ниже roadmap приведен в последовательности, соответствующей `PLAN.md`. Если часть функциональности уже существует в проекте, она сохраняется в плане, но помечается по фактическому состоянию относительно целевой архитектуры.

### **Фаза 1: Фундамент (PostgreSQL и базовый API)**

**Статус:** Частично реализовано.

*Цель: Создать структуру данных и базовые API загрузки/статусов, не включая векторы и LLM-поиск.*

- В проекте уже есть упрощенная модель `Document`, базовая загрузка PDF и список документов.
- Требуется перевести слой данных на целевую схему из `PLAN.md`: `documents`, `contract_facts` (`JSONB`), `extraction_runs`.
- Требуется внедрить `SQLAlchemy (async)` и миграции `Alembic`; текущий проект пока создает таблицы через `Base.metadata.create_all(...)`.
- `POST /documents/upload` должен работать как легкая ingestion-ручка: сохранять файл, создавать запись в БД со статусами `PENDING_REVIEW` / `QUEUED` и быстро возвращать ответ без парсинга в request cycle.
- Требуется `GET /documents/{id}` для отдачи текущих статусов и извлеченных фактов из PostgreSQL.
- Фаза считается завершенной, когда загрузка и polling статусов работают end-to-end без векторов и без LLM-зависимостей в синхронном API.

### **Фаза 2: Асинхронный воркер (Парсинг и извлечение)**

**Статус:** Не реализовано.

*Цель: Перенести тяжелую обработку PDF в фон и доводить документ до состояния `FACTS_READY` без индексации в Qdrant.*

- Настроить `Celery + Redis` для FastAPI и заменить мок/заглушку после `POST /documents/upload` на `process_document.delay(document_id)`.
- Реализовать фоновый lifecycle статусов: `QUEUED` -> `PARSING` -> `FACTS_READY` либо failure state с логированием ошибки.
- Добавить layout analysis через `DocLing` или эквивалентный парсер с сохранением Markdown-структуры, таблиц и page anchors.
- Добавить сервис извлечения фактов через LLM с `Pydantic`-валидацией ответа и записью JSON в `contract_facts`.
- Ввести структурированное логгирование для статусов документа, retries, ошибок парсинга, LLM и интеграций.
- Фаза считается завершенной, когда документ после загрузки обрабатывается воркером и стабильно доходит до `FACTS_READY`.

### **Фаза 3: Валидация и векторный слой (Qdrant)**

**Статус:** Частично реализовано.

*Цель: Сделать ручное подтверждение обязательным шагом и индексировать только подтвержденные данные.*

- В проекте уже есть упрощенная ручка подтверждения и базовая запись в Qdrant, но она не соответствует целевой архитектуре.
- Требуется `POST /documents/{id}/confirm`, которая обновляет JSON-факты в БД, переводит документ в `APPROVED` и запускает `index_document.delay(document_id)`.
- Требуется отдельный Qdrant-клиент и инициализация двух коллекций: `contract_summaries` и `contract_chunks`.
- Требуется реализовать idempotent-индексацию: генерация summary, нарезка Markdown на чанки, удаление старых векторов по `document_id`, запись новых в обе коллекции, перевод статуса в `INDEXED`.
- Индексация должна происходить только после ручного подтверждения менеджером; неподтвержденные факты не попадают в поиск.
- Фаза считается завершенной, когда подтверждение становится обязательной границей доверия, а переиндексация одного и того же документа не создает дублей.

### **Фаза 4: Оркестрация поиска (Query Router)**

**Статус:** Частично реализовано.

*Цель: Связать PostgreSQL и Qdrant через явную и тестируемую маршрутизацию запросов.*

- В проекте уже есть упрощенный `/chat` и single-collection RAG c опциональным фильтром по `document_id`, но отсутствует целевая маршрутизация.
- Требуется реализовать `Query Router`, который классифицирует запрос в маршруты вроде `SQL_ONLY`, `SUMMARY_ONLY`, `SUMMARY_THEN_CHUNKS`, `CHUNKS_ONLY`, `HYBRID_SQL_AND_CHUNKS` и извлекает структурные фильтры (`year`, supplier, date range и т.д.).
- Требуется SQL-сервис для ответов на строгие запросы по фактам и статусам поверх `contract_facts`.
- Требуется гибридный поиск `SUMMARY_THEN_CHUNKS`: сначала поиск по `contract_summaries`, затем поиск по `contract_chunks` только внутри top-N найденных `document_id`.
- Требуется обновить чат-оркестрацию так, чтобы ответы строились по выбранному маршруту, а не по одному универсальному retrieval path.
- Фаза считается завершенной, когда глобальный и локальный поиск идут по разным retrieval-стратегиям, а structured queries могут обходиться без векторного слоя.

### **Фаза 5: Автоматизация актов (Cron)**

**Статус:** Не реализовано.

*Цель: Внедрить контроль сроков и замкнуть цикл "Договор -> Акт".*

- Разработка UI для загрузки акта и привязки акта к договору.
- Интерфейс для менеджера: список документов «В работе» и состояние по закрывающим документам.
- Разработка `Celery Beat`/cron-сервиса: реализация SQL-логики светофора (15-10-5 дней) и рассылка email-уведомлений без участия LLM.
- Присвоение флага `OVERDUE` / "Просрочено" для договоров с истекшим дедлайном.
- Фаза считается завершенной, когда напоминания и overdue-статусы вычисляются детерминированно из PostgreSQL.

### **Фаза 6: Аналитика и управление (Dashboard)**

**Статус:** Не реализовано.

*Цель: Дать руководству инструменты контроля дисциплины и использования системы.*

- Разработка панели администратора / директора.
- Базовое управление доступом: добавление и удаление сотрудников, переназначение договоров, контроль ownership.
- Аналитический дашборд: агрегация по менеджерам, скорости закрытия, количеству открытых договоров и доле просрочек.
- Метрики adoption: контроль, что загрузка и обработка договоров действительно происходят через систему.
- Фаза считается завершенной, когда директор видит прозрачную картину просрочек, нагрузки и дисциплины по команде.
