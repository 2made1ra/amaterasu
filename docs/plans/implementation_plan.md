# Implementation Plan

Owner: Engineering
Verification: Draft
Tags: backend, async, ingestion, rag

**Status:** In Progress | **Audience:** Engineering Team, AI Agent, Migration Reviewers

## 1. Purpose

This document captures the practical implementation plan for the backend portion of hybrid contract search based on [PLAN.md](../../PLAN.md) and the additional task-decomposition rules for working with an AI agent.

This document is intentionally focused on phases 1-4:
- PostgreSQL and the basic API
- the asynchronous worker
- manual confirmation and Qdrant indexing
- search orchestration

The phases covering cron notifications and the executive dashboard should be moved into a separate plan after the ingestion and retrieval flow is stabilized.

## 2. Current Baseline In Repository

At the time this plan was prepared, the repository already contains a working but non-target baseline:

- `backend/app/api/api_v1/endpoints/documents.py` performs uploads synchronously and tries to run extraction and indexing inside the request cycle.
- `backend/app/models/document.py` stores only the simplified `PENDING` / `CONFIRMED` status pair, without an explicit lifecycle for ingestion, review, and indexing.
- `backend/app/services/rag.py` works with a single `contracts` collection rather than the two-layer `contract_summaries` and `contract_chunks` model.
- The project does not yet have the target `Celery + Redis` worker pipeline.
- The project does not yet have the target PostgreSQL schema for `documents`, `contract_facts`, and `extraction_runs`.
- The project already has a basic `/chat`, but it does not include a query router or a separation between SQL search and summary-first retrieval.

As a result, implementation should not start "from scratch." It should evolve the existing code carefully through small, reversible steps.

## 3. Hard Rules For Agent Work

### 3.1. Sequential execution only

- Do not give the agent the full specification at once.
- Give it only the current phase and only one sub-plan at a time.
- Do not move to the next sub-plan until the previous one:
  - works,
  - is covered by at least one basic test,
  - has passed manual review using the checklist below.

### 3.2. Logging is mandatory

- Require structured logging in all background tasks and services.
- Log every document status transition.
- Log Celery task start, retry, failure, and completion events.
- Separate errors by source: Postgres, PDF parsing, LLM, and Qdrant.
- Any sub-plan without logs on critical transitions must be considered incomplete.

### 3.3. Migrations require manual review

- Always read the generated SQL manually before applying an Alembic migration.
- Pay special attention to `drop table`, `drop column`, type changes, and implicit renames implemented as drop/create.
- Do not accept autogenerated migrations as correct without manual validation.

### 3.4. Minimal change surface

- Each sub-plan should modify only the layer required for the current goal.
- Do not touch Qdrant until Phase 2 is complete.
- Do not touch the query router until indexing is complete and stable.

## 4. Delivery Strategy

Each sub-plan must define the following:

- `Input`: which slice of `PLAN.md` and which context should be given to the agent.
- `Prompt`: the exact task wording for the agent.
- `Expected result`: what should exist after implementation.
- `Basic test`: the minimum automated test required for the step to count as complete.
- `Manual verification`: what must be checked by hand before moving on.
- `Exit gate`: the formal completion condition for the sub-plan.

## 5. Scope Boundaries

This document intentionally excludes:

- email notifications and contract-closing cron logic;
- the executive dashboard;
- full RBAC beyond the current ownership model;
- frontend polish beyond the API contracts required by the backend;
- production hardening beyond the local/dev rollout.

## 6. Phase Plan

### Phase 1. Foundation

**Goal:** create the PostgreSQL schema and lightweight upload/status API endpoints without vectors and without LLM-based search.

#### Phase 1.1. PostgreSQL schema

**Input for agent**

- From `PLAN.md`: the `Phase 1: Foundation` section.
- From the architecture context: only the PostgreSQL storage model.
- Do not provide anything about Qdrant, retrieval, or `/chat`.

**Prompt**

> Write SQLAlchemy (async) models and Alembic migrations for the `documents`, `contract_facts` tables using `JSONB`, and `extraction_runs`. Do not implement anything related to vectors or search.

**Implementation notes for this repository**

- Expand the existing document model into the target lifecycle.
- Introduce separate statuses at minimum for review, processing, and indexing.
- Do not rely on `Base.metadata.create_all(...)` as the primary schema mechanism.
- Prepare the schema to support multiple extraction runs per document.

**Expected result**

- The `documents` table stores file metadata, owner, current statuses, active extraction version, and last error.
- The `contract_facts` table stores extracted JSON facts and extraction versioning.
- The `extraction_runs` table stores processing attempts, timestamps, status, and error details.

**Basic test**

- A test for successful `alembic upgrade head`.
- A test confirming that the tables and key columns are created.

**Manual verification**

- Review the generated migration SQL.
- Confirm that there are no destructive operations without a valid reason.
- Check that relationships and indexes look predictable.

**Exit gate**

- The migration applies and rolls back locally.
- There is at least one schema-level test.

#### Phase 1.2. Upload API

**Input for agent**

- From `PLAN.md`: the lightweight ingestion section.
- Provide only upload behavior, without DocLing, without Celery, and without LLM extraction.

**Prompt**

> Implement `POST /documents/upload` in FastAPI. It must only save the file to disk/S3, create a database record with `PENDING_REVIEW` and `QUEUED`, and return `200 OK`. Mock asynchronous work for now with a placeholder function using `pass`.

**Implementation notes for this repository**

- Remove heavy PDF processing from the request cycle.
- Preserve compatibility with the current auth/deps chain.
- If S3 is not connected yet, use local file storage as the baseline.

**Expected result**

- Upload returns quickly and deterministically.
- A document record is created in the database with the initial statuses.
- The file is saved without attempting parsing or indexing inside the request.

**Basic test**

- An API test for `POST /documents/upload` with a PDF file.
- A check that the record is created and the statuses are set correctly.

**Manual verification**

- Confirm that the endpoint does not call RAG, LLM, or Qdrant code.
- Confirm that the file path is persisted correctly.

**Exit gate**

- Upload works end to end.
- Response time does not depend on PDF parsing.

#### Phase 1.3. Status API

**Input for agent**

- Only the document state and extracted facts read flow.

**Prompt**

> Implement `GET /documents/{id}` so it returns the current statuses and extracted facts from the database.

**Implementation notes for this repository**

- The response should support frontend polling.
- Explicit lifecycle and error fields are required.
- Return `404` for a missing document or a document owned by another user.

**Expected result**

- The endpoint returns document metadata, statuses, and facts.
- If facts do not exist yet, the response remains valid and predictable.

**Basic test**

- An API test for successful document retrieval.
- An API test for `404`.

**Manual verification**

- Confirm that the response is suitable for UI polling.
- Confirm that JSON facts serialize cleanly.

**Exit gate**

- A polling-ready API contract exists.
- Phase 1 works without the vector layer and without LLM dependencies in the synchronous API.

### Phase 2. Background Processing

**Goal:** move PDF parsing and fact extraction into a resilient asynchronous worker pipeline.

#### Phase 2.1. Celery + Redis wiring

**Input for agent**

- Only queue infrastructure and the worker entrypoint.
- Do not include Qdrant or query-routing details.

**Prompt**

> Configure `Celery + Redis` for FastAPI. Add a worker entrypoint. Replace the mock in `POST /documents/upload` with a call to the Celery task `process_document.delay(doc_id)`.

**Implementation notes for this repository**

- Define the Celery app, task module, and startup/dev instructions separately.
- Log the enqueue event and task id explicitly.
- Do not mix worker code into the request handler.

**Expected result**

- Upload enqueues a task.
- The worker can run independently from the API.
- Logs clearly show the path `QUEUED -> task received`.

**Basic test**

- A test confirming that upload calls `process_document.delay`.
- A smoke test for worker startup in the local environment.

**Manual verification**

- Check the Redis configuration and worker startup flow.
- Confirm that the API still accepts uploads if the worker is down.

**Exit gate**

- The queue is connected.
- Task invocation is asynchronous and visible in logs.

#### Phase 2.2. Layout analysis

**Input for agent**

- Only PDF parsing and layout extraction.
- Do not provide the full extraction prompt.

**Prompt**

> Write a service function that takes a PDF from disk, runs it through DocLing or an equivalent library, and returns Markdown while preserving tables. Integrate this into the Celery task, and make sure the status changes to `PARSING`.

**Implementation notes for this repository**

- Separate the parser service from task orchestration.
- The output must be traceable: either a Markdown artifact or an explicit storage/reference to the result.
- If an alternative to DocLing is chosen, record the reason in a docstring or docs.

**Expected result**

- The worker reads the PDF from disk and produces a Markdown representation.
- Tables and the main document structure are preserved as completely as possible.
- Parsing errors update status and write actionable diagnostic logs.

**Basic test**

- A service-level test using a sample PDF or a stubbed parser.
- A test that confirms failures move the document into a failure state.

**Manual verification**

- Inspect the Markdown generated from one real PDF.
- Confirm that tables do not collapse into unreadable text.

**Exit gate**

- Parsing runs in the worker.
- Logs exist for parsing start, completion, and failure.

#### Phase 2.3. Fact extraction

**Input for agent**

- Only the extraction pipeline on top of Markdown.
- Do not include vectorization or chunk indexing.

**Prompt**

> Write a fact extraction service. Use a Pydantic schema to validate the response from OpenAI. Input is Markdown from DocLing. Output is JSON. Save the JSON into `contract_facts` in Postgres and move the status to `FACTS_READY`.

**Implementation notes for this repository**

- Separate prompt construction, LLM client calls, and validation.
- Persist both the extraction run and the resulting facts.
- Treat validation errors as a separate class of errors rather than mixing them with transport/runtime failures.

**Expected result**

- Valid LLM output is stored in `contract_facts`.
- Invalid responses do not corrupt the database and leave a diagnostic trail.
- The document reaches `FACTS_READY`.

**Basic test**

- A test for successful validation and fact persistence.
- A test for an invalid LLM payload.

**Manual verification**

- Inspect the stored JSON for one real document.
- Check the status log `QUEUED -> PARSING -> FACTS_READY`.

**Exit gate**

- After upload, the document reaches `FACTS_READY` in the background.
- No Qdrant integration exists at this step yet.

### Phase 3. Approval And Vector Layer

**Goal:** make manual confirmation a mandatory trust boundary and index only approved documents.

#### Phase 3.1. Confirmation API

**Input for agent**

- Only review and approval behavior.
- Do not include the query router.

**Prompt**

> Implement `POST /documents/{id}/confirm`. It must update the JSON in the database if the manager edited it, move the document to `APPROVED`, and trigger a new Celery task `index_document.delay(doc_id)`.

**Implementation notes for this repository**

- Confirmation should operate on facts in `contract_facts`, not on the old `extracted_deadline` model.
- If the data model allows it, separate draft facts from the approved snapshot explicitly.
- The endpoint must not index the document synchronously.

**Expected result**

- A manager can confirm or edit extracted facts.
- After confirmation, the document becomes ready for indexing.

**Basic test**

- An API test for confirm with modified JSON.
- A test confirming that `index_document.delay` is called.

**Manual verification**

- Confirm that an unapproved document is not indexed.
- Confirm that the approved state is visible through `GET /documents/{id}`.

**Exit gate**

- Manual approval is required before indexing.

#### Phase 3.2. Qdrant setup

**Input for agent**

- Only the Qdrant client and collection initialization.
- Do not provide the full `index_document` logic.

**Prompt**

> Write a client class for Qdrant. Create initialization for two collections: `contract_summaries` and `contract_chunks` with the required vector settings.

**Implementation notes for this repository**

- Move away from the single `contracts` collection.
- Make collection configuration deterministic and repeatable.
- Initialization may happen on startup or before indexing, but the behavior must be idempotent.

**Expected result**

- There is one explicit access layer for Qdrant.
- Summary retrieval and chunk retrieval are physically separated.

**Basic test**

- A unit test for collection configuration and the ensure-collection flow.

**Manual verification**

- Confirm locally that both collections exist.
- Check distance and vector-size settings.

**Exit gate**

- Qdrant is ready for two-layer indexing.

#### Phase 3.3. Idempotent indexing

**Input for agent**

- Only the approved indexing pipeline.
- Provide the summary-generation prompt separately from the query-routing logic.

**Prompt**

> Implement the Celery task `index_document`. Step 1: generate a summary with the LLM. Step 2: split the Markdown into chunks. Step 3: delete old vectors by `document_id`. Step 4: write new vectors into both collections. Move the status to `INDEXED`.

**Implementation notes for this repository**

- Indexing must use approved data only.
- Summaries and chunks are indexed separately, but within one orchestration flow.
- Deleting old points before writing new ones is mandatory for idempotency.

**Expected result**

- The document receives summary embeddings and chunk embeddings.
- Re-running indexing does not create duplicates.
- The status reaches `INDEXED`.

**Basic test**

- A test confirming old vectors are deleted before upsert.
- A test confirming writes happen to both collections.

**Manual verification**

- Re-index the same document twice.
- Confirm that the number of vectors does not grow uncontrollably.

**Exit gate**

- Approved-only indexing works.
- Re-indexing is safe.

### Phase 4. Query Orchestration

**Goal:** connect PostgreSQL and Qdrant through an explicit router instead of a single generic retrieval path.

#### Phase 4.1. Query router

**Input for agent**

- From `PLAN.md`: only the routing and retrieval block.
- Do not provide the old RAG implementation in full.

**Prompt**

> Write a service that uses a lightweight LLM or hard logic to classify a user query into one of 5 types: `SQL_ONLY`, `SUMMARY_ONLY`, `SUMMARY_THEN_CHUNKS`, `CHUNKS_ONLY`, or `HYBRID_SQL_AND_CHUNKS`. Return both the route type and extracted filters, for example `year=2023`.

**Implementation notes for this repository**

- The router should be testable and deterministic whenever possible.
- Use an explicit typed result rather than a "magic string."
- Extracted filters must be usable by SQLAlchemy.

**Expected result**

- Every query receives a route decision.
- Structured filters are extracted separately from retrieval.

**Basic test**

- Table-driven tests for several classes of queries.

**Manual verification**

- Check 10-15 typical team queries by hand.
- Confirm that the router does not send everything into vector search.

**Exit gate**

- Query classification is stable and explainable.

#### Phase 4.2. SQL search

**Input for agent**

- Only the SQL path.
- Do not mix it with chunk search.

**Prompt**

> Write a `SQL_ONLY` search service that generates a SQLAlchemy query against the `contract_facts` table based on the extracted filters.

**Implementation notes for this repository**

- Rely on JSONB fields and explicit filters.
- Handle empty results and invalid filters.

**Expected result**

- Structured questions can be answered without Qdrant.
- The reporting path becomes reproducible and inexpensive.

**Basic test**

- Tests for filtering by year, supplier, and date range.

**Manual verification**

- Manually compare one or two SQL results against the data in the database.

**Exit gate**

- `SQL_ONLY` works independently of the vector layer.

#### Phase 4.3. Hybrid vector search

**Input for agent**

- Only the summary-first retrieval flow.
- Do not overload the prompt with SQL-path details.

**Prompt**

> Write a vector search service. First search the `contract_summaries` collection. Take the top 3 `document_id` values. Then search the `contract_chunks` collection, strictly filtering by those three `document_id` values.

**Implementation notes for this repository**

- Summary search is required for global discovery.
- Chunk search must be tightly constrained to the selected documents.
- The result contract for chat orchestration should be clear and explicit.

**Expected result**

- Relevant documents are selected first.
- Deep search then happens only inside that shortlist.

**Basic test**

- A test confirming that chunk search receives a filter based on the shortlisted `document_id` values.

**Manual verification**

- Compare behavior with single-stage retrieval on a couple of difficult questions.
- Confirm that top-3 actually limits the second step.

**Exit gate**

- Hybrid retrieval works as a clear two-step flow and does not fall back to a global chunk search.

## 8. Suggested Test Matrix

At least one basic test is required for each sub-plan:

- `1.1` schema migration smoke test
- `1.2` upload API integration test
- `1.3` document status read test
- `2.1` enqueue test for Celery task dispatch
- `2.2` parser service test
- `2.3` facts extraction validation test
- `3.1` confirm API test
- `3.2` Qdrant collection setup test
- `3.3` idempotent indexing test
- `4.1` query router classification test

## 9. Human Review Checklist Before Each Next Step

- The previous step genuinely works end to end, not just "compiles."
- There is at least one automated test for the step's critical scenario.
- Logs show status transitions and error causes.
- Migrations were read manually if the step changed the schema.
- The new step did not pull in dependencies from future phases too early.

## 10. Definition Of Done For This Plan

This plan is complete when:

- `POST /documents/upload` no longer performs heavy processing synchronously.
- Document ingestion moves through explicit statuses and background tasks.
- Extracted facts are persisted to PostgreSQL before any indexing begins.
- Manual confirmation is required before writing to Qdrant.
- Qdrant uses two collections: `contract_summaries` and `contract_chunks`.
- Re-indexing by `document_id` is idempotent.
- The query router controls the choice between SQL, summaries, and chunk retrieval.
- Every sub-plan is implemented and covered by at least one basic test.
